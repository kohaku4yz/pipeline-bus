# Task 000 — 示例任务单(随便改,这只是格式范例)

- target_repo: <owner>/<repo>(或"无",看 mode)
- mode: **patch**(产出放 `work/000/`,附 `APPLY.md` 由 Owner 端落地) / `direct`(实现端直接 clone 目标 repo 开 branch,仅适用于你有写权限的仓库)

## 标题

一句话说清要做什么。带任务编号。

## 需求

具体描述要做的东西。可以分点写。每点要细到实现端**不用再问你**就能动手。

写需求时的几条经验:

- **给边界条件**: "同一天多次跑结果要一致"、"列表为空时不报错"、"用户名为空字符串时 fallback 到 'anonymous'"
- **给数据/格式样例**: 别只说"输出 JSON",给一个 example payload
- **指明风格/约定要保持一致**: 如果目标代码库用某种命名 / 缩进 / 错误处理风格,明确说"风格与现有代码一致"
- **标明 anti-feature**: "不要做 X" 这种比"只做 Y"更不容易扯皮

## 验收标准

**所有验收都要可机械判定**(跑一个命令看结果)。用 checkbox:

- [ ] `bash -n work/000/foo.sh` 无报错(parse 通过)
- [ ] `python3 work/000/foo.py --self-test` 全部用例通过
- [ ] 关键功能跑一遍有符合预期的 stdout 输出
- [ ] `work/000/self-test.txt` 已附自测输出截录
- [ ] 现有命令 / 函数 smoke test 无回归(列出要测的命令)
- [ ] 产出: `work/000/<产物文件>` + `work/000/APPLY.md` + 可选 `work/000/NOTES.md`

**反面例子**(别这么写,审查官没法机械判定):

- ❌ "代码质量要好"
- ❌ "性能要足够好"
- ❌ "风格要符合现有代码"

这些都该换成具体的:**"在 1000 行 input 下,跑 < 2 秒"**、**"用 4 空格缩进,命名用 snake_case"**。

## 落地相关(只对 `mode: patch`)

如果是 patch 模式,在需求里说清楚:

- 目标仓库路径(或 `<owner>/<repo>` GitHub 标识)
- 是新增文件还是改文件(改的话提供参考代码位置)
- Owner 落地后要不要跑特定命令实测

直接给出 `git show origin/task/NNN:work/NNN/<file>` 这类命令预期,实现端照写 APPLY.md。

## 实现端自由发挥 / 提建议的空间

如果你不 care 实现端具体怎么写,**把需求列清楚就够了**。比如"加个 `/api/foo` GET 接口,接受 `?id=...`,返回 JSON `{"id": "...", "ts": ...}`" — 路由怎么命名、错误码怎么定、参数校验到多严,实现端自己看着办,但在 `NOTES.md` 里写明每个决定。

如果你**特别 care**某几个细节,把那几个写死: "一定要用 FooBar 函数处理 X"、"绝对不要碰 Y 字段"、"错误时返回 422 不是 400"。

---

## 给实现端的话(可选)

任务单可以加一段私聊语气的话,效果很好: "刚踩过这个坑,别再走了"/"这个我已经试过两种方案,选 A"/"今天心情好,随意发挥" —— 让实现端知道你想要什么样的 vibe。

但别让它成为硬性要求 —— 决定仍然看**客观的验收标准**。
